# 第 23 周：组合监控、模型治理与应急演练

## 学习目标（本周目标）

生产系统的可靠性不能靠“代码应该没问题”证明，而要通过故障注入验证检测、停止、证据保存、回滚和恢复。本周覆盖数据、模型、风险与账户五类以上故障。完成后应能：

1. 将数据质量、PSI、回撤和账户对账统一成结构化停止决策。
2. 区分自动停止、回滚与恢复，理解三者不是同一个动作。
3. 为每类故障定义负责人、SLA、通知对象和恢复证据。
4. 通过桌面演练测量发现时间、停止时间和恢复时间。
5. 坚持“安全停止可自动，恢复必须人工批准”的治理边界。

## 前置知识

需要理解行情主键 `(date, symbol)`、净值回撤、PSI、持仓与现金对账。还应掌握第 21 周的降级流程和第 22 周的订单撤销含义。

## 核心理论

本周核心理论把数据校验、模型漂移、风险阈值和账户对账统一为可审计决策，并把停止、回滚、恢复拆成不同权限与证据门槛。演练目标不是证明不会故障，而是证明故障时能进入安全终态。

## 理论一：监控是决策系统

监控不只是画图。它接收可验证输入，输出：

- `should_stop`：是否停止。
- `reasons`：去重后的结构化原因码。
- `metrics`：用于审计的数值，例如最大回撤和 PSI。

对净值序列 \(NAV_t\)，代码计算

\[
DD_{\max}=\max_t\left(1-\frac{NAV_t}{\max_{s\le t}NAV_s}\right).
\]

当 `DD_max >= max_drawdown` 时原因是 `MAX_DRAWDOWN`。PSI 非有限或达到阈值时原因是 `PSI_DRIFT`；账户不匹配是 `ACCOUNT_MISMATCH`；行情异常或非法净值是 `DATA_ANOMALY`。默认阈值为最大回撤 15%、PSI 0.25，但本周模型漂移演练使用 PSI 0.10。

## 理论二：数据故障检测

`validate_market_data` 依次验证：

1. 必需列 `date`、`symbol`、`close`、`volume` 完整。
2. 表非空，日期可解析。
3. `(date, symbol)` 不重复。
4. `close` 有限且大于 0。
5. `volume` 有限且不小于 0。
6. `symbol` 非空。
7. `as_of` 不早于最新日期，且数据年龄不超过门槛。

本周注入三种数据故障：

- `MISSING_COLUMN`：删除 `volume`。
- `DUPLICATE_ROW`：复制第一行，制造重复主键。
- `STALE_DATA`：把观察时点推进 5 日，而允许最大年龄为 1 日。

捕获到 `ValueError` 后，停止规则以 `data_error` 触发 `DATA_ANOMALY`。生产实现应保留异常类别、数据批次、校验时间和受影响范围；本周 CSV 只保留合并后的原因码。

## 理论三：模型、风险与账户故障

`MODEL_DRIFT` 把基准 \([-1,1]\) 与当前 \([2,4]\) 比较，PSI 应显著超过 0.10，回滚到上一批准模型。

`MAX_DRAWDOWN` 使用净值 `[1.0, 0.96, 0.80]`。峰值为 1，末值为 0.8，最大回撤为 20%，超过 15% 阈值；动作是撤销未成交订单、冻结调仓、人工复核。

`ACCOUNT_MISMATCH` 比较预期持仓 AAA=1000 与实际 900、预期现金 100000 与实际 99000。默认股数容差 0、现金容差 0.01，因此两者均不匹配。账户不一致时不能继续用策略内部账本下单，否则可能重复买卖或突破风险限额。

## 理论四：停止、回滚、恢复

三者必须分开：

- 停止：阻止风险继续扩大，包括停止新订单、撤销未成交单。
- 回滚：恢复上一已验证数据快照、批准模型或确认账本。
- 恢复：在根因关闭后重新开启运行。

停止可以由明确、保守、幂等的规则自动执行。恢复需要确认故障不是暂时消失，必须经过离线重放、影子运行、第二人复核和人工批准。回滚也不是简单覆盖：必须先保存故障现场，否则会丢失审计证据。

## 阅读顺序

先逐行核对故障检测与停止原因，再按手册演练停止—保存证据—回滚—恢复，随后用摘要复核覆盖率，最后补齐责任、时效和批准记录。

## 实验步骤

1. 先读 `fault_injection.csv`，确认六行故障：三种数据、一种模型、一种回撤、一种账户。
2. 对每行检查 `detected` 与 `should_stop`，正常应全部为真。
3. 将 `reason` 映射到四类标准原因码，核对 `rollback` 是否与故障域匹配。
4. 读 `stop_and_rollback.md`，按停止顺序做桌面演练。
5. 用 `summary.csv` 核对故障数、检测数、停止数。
6. 补充原产物没有显式保存的责任人、SLA、通知与时间戳，形成作业证据。

## 逐文件与字段解释（逐产物与字段字典）

### `fault_injection.csv`

- `fault`：`MISSING_COLUMN`、`DUPLICATE_ROW`、`STALE_DATA`、`MODEL_DRIFT`、`MAX_DRAWDOWN` 或 `ACCOUNT_MISMATCH`。
- `detected`：故障是否被检测。本实验六类都应为真。
- `should_stop`：集中停止规则是否要求停止。
- `reason`：以 `|` 连接的原因码；数据故障为 `DATA_ANOMALY`，其余对应 PSI、回撤、账户原因。
- `rollback`：该故障的回滚动作文本。

当前演练中每行通常只有一个原因；真实事故可能同时出现多个原因。原因码应用于机器处理，`fault` 用于描述注入场景，两者不要混淆。

### `stop_and_rollback.md`

停止顺序：

1. 停止生成新订单并撤销尚未成交订单。
2. 保存行情批次、模型版本、账户状态和结构化告警。
3. 按故障类型恢复上一份已验证数据、批准模型或确认账本。

恢复门槛：

- 根因、影响范围与修复证据由第二人复核。
- 离线重放和影子运行均通过，停止规则不再触发。
- 必须人工批准，不因告警自动消失而自动重启。

这份手册是最小模板，生产版还应有负责人、升级路径、券商联络、RTO/RPO、命令与权限边界。

### `summary.csv`

- `fault_types`：注入故障行数，应为 6。
- `detected_faults`：`detected` 为真的数量，应为 6。
- `stop_decisions`：`should_stop` 为真的数量，应为 6。

三个数字相等只证明预设案例通过，不证明未知故障都能覆盖。

### `homework.md`、`acceptance.json`、`manifest.json`

作业要求补齐责任和时效并执行桌面推演；验收要求至少五类故障均具备检测、停止和回滚，且恢复需人工批准；manifest 声明本次周数、模式和产物路径。

本周没有强制图形产物。建议从 `fault_injection.csv` 画“故障×阶段”矩阵，列出检测、停止、回滚、恢复证据，缺失项用醒目颜色标注；桌面演练另画时间线，标出注入、发现、停止、根因关闭、影子运行和恢复批准时刻。矩阵全绿只表示预设案例通过；时间线中的长间隔才揭示 MTTD、停止延迟和 MTTR 瓶颈。

## 动态指标判断

- 生产监控应滚动观察数据年龄、异常批次数、PSI、最大回撤和对账差额，并记录阈值、连续次数、首次触发时间和原因码。
- 硬安全条件（非法数据、账户不一致）单次触发即可停止；统计指标可采用连续窗口或滞回阈值，但规则必须预先批准。
- 同一时刻多个原因触发时保留全部原因，不让后一个告警覆盖前一个；优先执行共同的安全停止，再按证据链调查根因。
- 告警消失不等于恢复。只有离线重放、影子运行、第二人复核和人工批准全部完成，状态才从“停止”转为“可恢复”。

## 故障响应判断表

### 数据异常

立即拒绝新批次，继续使用上一已验证快照只适用于快照仍在时效范围且业务批准的情况。若无法证明数据新鲜和完整，应停止交易。

### 模型漂移

停止新模型打分，切到预先批准的冻结规则组合或现金。保留输入分布、模型版本、特征代码和最近预测；只有 PSI 恢复并不够，还需影子绩效证据。

### 最大回撤

先阻止新增风险，避免故障中“自动抄底”。确认回撤来自市场、执行、数据还是账本；恢复仓位应经过风险批准。

### 账户不一致

以券商/模拟账户的已确认事实为外部核对源，冻结账户相关动作，逐笔对账。不能未经调查就让任一侧覆盖另一侧。

## 常见误区

1. 告警只发消息，不连接停止动作。
2. 停止后立即覆盖坏数据或模型，破坏现场证据。
3. 把回滚等同恢复，回滚完成就自动开机。
4. 只测单一故障，不测多故障并发和告警去重。
5. 测试检测逻辑，却不实际演练撤单和权限。
6. 只记录“成功恢复”，不记录发现、响应和恢复时间。
7. 负责人写“团队”，没有值班角色和升级对象。
8. 因误报担忧放宽硬停止线，却没有分级告警设计。

## 思考题

1. 为什么数据过期和账户不一致即使策略收益良好也应停止？
2. 自动回滚上一模型时，如何证明上一模型及其依赖仍可运行？
3. 停止规则同时触发 `PSI_DRIFT` 和 `MAX_DRAWDOWN` 时，根因调查顺序应如何安排？
4. 哪些停止动作必须幂等？重复调用撤单可能有哪些边界情况？
